Συμβουλές για το Git που θα ήθελα να γνωρίζω από το πρώτο έτος

  • Post author:geoapos
  • Post published:May 10, 2022
  • Post category:Uncategorized
  • Post comments:0 Comments

Καλησπέρα, καλώς ήρθατε στο σημερινό πρώτο μάθημα για το Git – Github – Gitlab. Θα κάνουμε μία εισαγωγή στο Git, τι ακριβώς είναι, πως χρησιμοποιείται. Αρχικά θα ασχοληθούμε με το terminal και στη συνέχεια με γραφικά περιβάλλοντα όπως είναι το GitKraken ή το Github Desktop. 

Ξεκινώντας, το Git αποτελεί ένα software για να κρατάμε εκδόσεις αρχείων. Κατά 99,9% χρησιμοποιείται για κώδικα και η βασική λογική πίσω από αυτό είναι ότι κρατάει snapshots του κώδικα κατά τη διάρκεια του χρόνου μέσα στα λεγόμενα Repositories. Το καλό με αυτά είναι ότι μπορούμε να δούμε την ιστορία του κώδικα μας, ποιος έχει κάνει αλλαγές, πότε και που. Ακόμα, σε περίπτωση που έχουμε βρεθεί σε κάποιο μεγάλο bug που επηρεάζει το production μπορούμε κάλλιστα να γυρίσουμε πίσω σε προηγούμενο version χωρίς να έχουμε κάποιο πρόβλημα. Χωρίς αυτό, θα έπρεπε να αποθηκεύουμε το project μας σε φακέλους κάθε φορά που τελειώναμε κάποιο session. Αυτό όπως καταλαβαίνουμε δεν είναι πρακτικό, και είναι πάρα πολύ αργό ειδικά όταν στο ίδιο project πρέπει να ασχολούνται περισσότερο από ένα άτομα. Υπάρχουν 2 κατηγορίες Version Control System, Centralized και Distributed. Στο πρώτο, όλοι μπαίνουν στον κεντρικό σέρβερ για να έχουν την τελευταία version του project. Μερικά παραδείγματα αποτελούν το Subversion και το Team Foundation Server. Το πρόβλημα είναι ότι αν ο σερβερ πέσει, δεν μπορούμε να κάνουμε καμία αλλαγή η να αλληλεπιδράσουμε  όσο είναι κάτω. Στο δεύτερο, το κάθε μέλος της ομάδας έχει ένα copy από το project στη δικιά του μηχανή και κάνει αλλαγές τοπικά. Αν ο κεντρικός σέρβερ πέσει μπορούμε να συχρονίσουμε τη δουλειά μας απευθείας με άλλα άτομα της ομάδας. Τέτοιο παράδειγμα αποτελεί το Git που θα δούμε σήμερα, και το Mercurial. Γιατί όμως χρησιμοποιεί όλος ο πλανήτης το git? Είναι δωρεάν, είναι open source, είναι εξαιρετικά γρήγορο και scalable. Αποτελεί ένα από τα βασικά requirements για οποιαδήποτε Software Engineering δουλειά, γιατί ακριβώς πάρα πολλές εταιρείες το χρησιμοποιούν. 

Αρκετά όμως με τα θεωρητικά, πάμε να δούμε πρακτικά κάποια βασικά πράγματα για το Git. Θα ξεκινήσουμε όπως αναφέρθηκα και νωρίτερα με το terminal. Πρέπει αρχικά, να εγκαταστήσουμε το git στις μηχανές μας. Στα mac ο πιο εύκολος τρόπος να το εγκαταστήσουμε είναι μέσω Command Line Tools η το Xcode, σε linux γράφουμε sudo apt-get install git, yum και zypper ανάλογα το λειτουργικό σύστημα που χρησιμοποιούμε. Συγκεκριμένα στο δικό μου μηχάνημα με openSUSE-Tumbleweed θα γράψω sudo zypper install git. Αν βρισκόμαστε σε windows μπαίνουμε στη σελίδα https://gitforwindows.org κατεβάζουμε το installation και το τρέχουμε.

Δημιουργούμε τον φάκελο που θέλουμε για το project μας, και είτε σε terminal μέσω του VSCode είτε μέσω του φακέλου η πρώτη κίνηση που θα κάνουμε είναι το git init, δηλαδή θα κάνουμε initialize τον φκελο αυτό σαν ένα Git Repository, βάζοντας αυτόματα στον φάκελο μας έναν φάκελο .git και παρακολουθώντας τον φάκελο για τυχών αλλαγές. Ας φτιάξουμε λοιπόν 2 αρχεία , ας πούμε index.html και style.css (touch index.html, touch style.css). Γράφοντας το git status θα μας δείξει ότι έχουμε 2 αρχεία στο repository μας τα οποία είναι Untracked, δηλαδή δεν έχουμε πει στο git τι να κάνει με αυτά τα αρχεία. Πρέπει λοιπόν να κάνουμε git add . Και αν ξαναγράψουμε git status θα δούμε ότι πλέον προστέθηκαν και είναι Staged. Η . δείχνει ότι επιλέγουμε όλα τα αρχεία από το τωρινό μας directory που είναι έτοιμα να προστεθούν. Αν θέλουμε κάποιο συγκεκριμένο αρχείο να κάνουμε add η εντολή είναι αυτή git add [file]. Το Staged σημαίνει λοιπόν ότι εμείς θέλουμε να στείλουμε τα αρχεία αυτά στο git, ωστόσο δεν έχει γίνει ακόμα η αποστολή. Για να γίνει λοιπόν η αποστολή αυτή πρέπει να γράψουμε την εντολή git commit -m “Initial Commit” με οποιοδήποτε μήνυμα θέλετε εσείς, συνήθως βάζουμε αυτό. Θα μας βγάλει λοιπόν αποτέλεσμα ότι δεν έχουμε ποιοί είμαστε, οπότε πρέπει να του δώσουμε ένα email και ένα username. Git config user.email “email” και git config user.name “username”. Πλέον είμαστε έτοιμοι να κάνουμε commit με το προηγουμενο command. 

Έστω λοιπόν ότι τώρα θέλουμε να κάνουμε μία αλλαγή. Αν πάμε και γράψουμε το git status θα δούμε ότι το αρχείο μας είναι labeled σαν modified. Μπορούμε να πατήσουμε git diff [file] για να δούμε ποιές είναι οι αλλαγές που έγιναν. Τώρα θέλουμε αυτές οι αλλαγές να αποθηκευτούν στο βασικό μας repository. Για να το αποθηκεύσουμε λοιπόν πρέπει να κάνουμε git add index.html και git commit -m “Added new Line”. Έστω ότι κάνουμε δεύτερη αλλαγή και είναι λανθασμένη. Αν θέλουμε να γυρίσουμε πίσω σε προηγούμενη κατάσταση, γράφουμε git checkout [file].

 Ένα από τα ποιο σημαντικά πράγματα στο git είναι τα branches. Ο λόγος που χρησιμοποιούμε branches ειδικά αν δουλεύουμε σε ομάδα είναι επειδή θέλουμε να μη δουλέψουμε στο κώδικα που δουλεύουν οι υπόλοιποι, να απομονοθούμε, να κάνουμε ότι είναι να κάνουμε, και όταν σιγουρευτούμε για το αποτέλεσμα να το στείλουμε στο κεντρικό branch. Εδώ που βρισκόμαστε, θα κάνουμε git checkout -b dev (το όνομα του καινούργιου branch). Τώρα μπορούμε να κάνουμε αλλαγές στο αρχείο και να το δούμε και το προσθέτουμε στο repository με το git status.  Έστω λοιπόν ότι αυτή η αλλαγή είναι σωστή και θέλουμε να τη στείλουμε στο κεντρικό master branch. Git checkour master και git merge dev και βλέπουμε ότι η αλλαγή που κάναμε στο dev ήρθε στο master branch.

Κάνουμε καινούργιο commit αφού κάνουμε αλλαγές που θεωρητικά είναι λάθος. Git add . , git commit -m “addeddd new”. Τώρα, αν θέλουμε να διορθώσουμε το commit, git commit –amend και πλέον ανοίγει ένας editor, διορθώνουμε το λάθος και το σώζουμε.

Τώρα, ας πούμε ότι θέλουμε να δημιουργήσουμε ένα νέο αρχείο script.js (touch script.js). Αν δούμε στο στάτους, θα μας πεί ότι είναι untracked, και όταν το προσθέσουμε βλέπουμε ότι λέει staged. Για κάποιο λόγο, δεν θέλαμε να γίνει staged, το βγάζουμε παντώντας git rm –cached script.js. Και στο status θα λέει πάλι untracked. 

Ένα από τα σημαντικότερα πράγματα στο git, και κάτι που άργησα πάρα πολύ να το μάθω και είχα θέμα, είναι το git ignore. Υπάρχουν κάποια project που χρησιμοποιούν αρχεία τα οποία εμείς δεν θέλουμε να φαίνονται παραέξω. Μερικά παραδείγματα είναι api tokens, πχ στο MongoDB η ακόμα και για ένα Discord Bot χρειάζεται να γραφτεί μέσα σε ένα αρχείο πχ Javascript ένα μοναδικό token. Αν δοκιμάσετε να στείλετε στο git αυτό το token, τότε θα σας έρθει email είτε από το API είτε από το Github ότι κάποιο token είναι exposed. Για να μην δείξουμε λοιπόν το αρχείο που περιέχει τέτοιου είδους δεδομένα (πχ .env files), δημουργούμε ένα αρχείο με ονομασία .gitignore. Μέσα εδώ  θα γράψουμε την ονομασία του αρχείου που δεν θέλουμε να φαίνεται. Στη συνέχεια κάνουμε commit και βλέπουμε ότι δεν είναι tracked το .env και δεν μας το ζητάει στη συνέχεια. 

Στη συνέχεια, θα δούμε το tag. Αυτό το κάνουμε για να πάρουμε ένα συγκεκριμένο snapshot του κώδικα και να του δώσουμε ένα όνομα, κάθε φορά που κάνουμε deploy, η σε κάθε release. Git tag v0.0.1 και git tag.

Καλά λοιπόν όλα αυτά, αλλά πρέπει να θυμόμαστε ότι το Git είναι για ομάδες. Πρέπει με κάποιο τρόπο να μοιράσουμε το κώδικα στους υπόλοιπους προγραμματιστές της ομάδας. Πάμε λοιπόν στο github και φτιάχνουμε το repository. Ένα πολύ καλό πράγμα είναι ότι βοηθάει το github αλλα πρέπει να κατεβάσουμε και εμείς 2 πραγματάκια ακόμα γράφοντας στο terminal sudo apt-get install ca-cerificates ssh-client. Επιλέγουμε το remote add origin, και το βαζουμε στο τερμιναλ, οπότε πλέον οποιαδήποτε αλλαγή κάνουμε θα ανεβαίνει σε αυτό το repository. Και τώρα θεωρητικά θα μπορούμε να κάνουμε git push -u origin master. Ωστόσο θα δούμε ένα access deny. Αυτό που πρέπει να κάνουμε είναι να φτιάξουμε ένα ssh key, pair, ενα secret και ένα public, ένα για εμάς και ενα για το Github. Για να γίνει αυτό κάνουμε ssh-keygen -t rsa -b 4096 -C “email”. Δεν βάζουμε κάτι από αυτά. Πάμε λοιπόν να βρούμε το public key. Με την εντόλη cat [path] το βλέπουμε, το αντιγράφουμε και το βάζουμε στο account μας, settings, ssh και gpg keys, και θα πατήσουμε new ssh key, paste και δίνουμε όνομα. Τώρα αμα κάνουμε push θα το δεχτεί κανονικά. 

Πάμε να δούμε τώρα το pull origin master για να πάρουμε τις αλλαγές που έχουν κάνει άλλοι προγραμματιστές. Ακόμα, μπορούμε να κάνουμε push και το tag που λέγαμε προηγουμένως, git push origin tag v0.0.1. Git fetch origin –prune 🡪 κατεβάζει όλα τα branches που μπορουν να υπάρχουν στο remote και σβήνει όλα αυτά που έσβησαν και οι άλλοι προγραμματιστές, παρόλο που χρησιμοποιούμε το fetch πρέπει μετά να πάμε σε κάθε branch και να κάνουμε pull. 

Pull Request: Ένα μεγάλο κεφάλαιο στο git και εξαιρετικά χρήσιμο, ήδη μεγάλες εταιρείες το χρησιμοποιούν καθημερινά. Ουσιαστικά η λογική είναι ότι θέλουμε να συμβάλουμε στον κώδικα κάποιου άλλου, αλλά πολλές εταιρείες το κάνουν έτσι ώστε να περναέι από τους Senior Developers έτσι ώστε να παίρνει έγκριση για το τελικό push. Σε αυτή τη περίπτωση, πάμε στο repository που θέλουμε να πειράξουμε, πατάμε fork, το βάζουμε σε εμάς, και θα αφήσουμε το github να φτιάξει το repository που ανήκει σε εμάς. Το κατεβάζουμε στο πισι μας, κάνουμε clone with ssh, copy και paste στο τερμιναλ ( σε διαφορετικο φακελο ). Τον ανοίγουμε, μπαίνουμε και απο το terminal, κάνουμε μία αλλαγή  και ας το κάνουμε commit. Θα δούμε ότι πρέπει να κάνουμε το config που λέγαμε προηγουμένως με το email και το username. Τώρα θα γίνει κανονικά το κομιτ, και θα κάνουμε και πους. Δεν βάζουμε ρεμοτε εφόσον κάνουμε clone απευθείας από πάνω. Πηγαίνουμε τώρα στο πρώτο ρεποσιτορυ, οχι δλδ στο δικο μας, και πάμε στα pull requests. New pull request, compare across forks, και επιλεγουμε το repository μας, φαίνεται η αλλαγή, και πατάμε pull request. Τώρα εδώ, πρέπει να γράψουμε διάφορα πράγματα. Γιατί κάναμε τις αλλαγές, τι ακριβώς λύνει αυτό που φτιάξαμε κτλ. Τώρα, θα πάει ο μαιντεινερ, θα δει ότι κάποιος έχει ενα πουλ ρικουεστ, και θα πει αν εγω θελω να το περασω στον κωδικα η οχι. Εμεις θα κάνουμε merge και οι αλλαγες περαστήκανε. Αν παμε τωρα στον κώδικα, θα δούμε ότι έχει μπει κανονικά η αλλαγή μας. 

Αρκετά όμως με το terminal, κουραστήκαμε. Έχουμε 2022 και υπάρχουν εφαρμογές που κάνουν τα πράγματα πιο ωραία, κατά τη γνώμη μου, και πιο έυκολα για τον καθημερινό χρήστη, χωρίς να χρειάζεται να πατάει καμία εντολή. 2 εφαρμογές που καλύπτουν γραφικά το περιβάλλον του git είναι το Github Desktop, που είναι η εφαρμογή του Github και λειτουργεί κυρίως σε Windows συστήματα, αλλά και το πιο ωραίο, κατά τη γνώμη μου, Gitkraken, που υπάρχει σε όλα τα συστήματα. Μπορείτε να το κατεβάσετε από το GitKraken.com. 

Θα δούμε λοιπόν σε εικόνα το user interface του για να μην χαθούμε. Στο αριστερό πάνελ θα βρούμε το gitflow, local, remote, stashes, pull requests, issues, tags, submodules και github actions. Πάνω, μπορούμε να βρούμε τα φίλτρα για το left panel, Repository/Branch Navigation, Open/Clone/Init, Tab, New Tab, το Toolbar στη μέση. Δεξιά θα δούμε τη λίστα με τα Tabs, notifications, preferences, settings για λογαριασμό. Δεξιά, υπάρχει το commit search, command palette και το Commit Panel, stage/unstage changes, view commit details, resolve conflicts και view files. 

Command Palette: Χρησιμοποιήστε το “Command/Control P” για να αλλάξετε το Command Palette: μια εισαγωγή κειμένου όπου μπορείτε να πληκτρολογήσετε εντολές για να ανοίξετε repositories, να προβάλετε το ιστορικό των αρχείων στον τρέχοντα χώρο αποθήκευσης και πολλά άλλα.

Μπορείτε επίσης να εκτελέσετε ενέργειες στο τρέχον αποθετήριο σας. Για παράδειγμα, μπορείτε να ανακτήσετε όλα, να δημιουργήσετε ή να εμφανίσετε μια αποθήκευση, να αναιρέσετε ή να επαναλάβετε ενέργειες κ.λπ. από την Παλέτα εντολών. Καθώς πληκτρολογείτε, οι σχετικές εντολές θα εμφανίζονται σε μια λίστα. Αυτό σας επιτρέπει να εκτελέσετε πολλές ενέργειες χωρίς να χρειαστεί ποτέ να αφήσετε την άνεση του πληκτρολογίου σας.

Για παράδειγμα, εάν θέλετε να ανοίξετε μια καρτέλα GitKraken Terminal, πληκτρολογήστε: “terminal” και επιλέξτε είτε “Open this repo in a Terminal tab” είτε “Open a new Terminal Tab” από τις επιλογές που εμφανίζονται.

Λειτουργία σε άλλα branches

Μία από τις πιο λεπτές δυνατότητες του GitKraken Client είναι η δυνατότητα αλληλεπίδρασης με branches (και tags) διαφορετικούς από αυτόν που ελέγχεται αυτήν τη στιγμή. Μπορείτε να φέρετε, να τραβήξετε και να σπρώξετε ένα branch ενώ βρίσκεστε σε διαφορετικό branch, υπό την προϋπόθεση ότι δεν υπάρχουν merge conflicts.

Για παράδειγμα, εάν έχετε ελέγξει τον branch “development”, μπορείτε να κάνετε δεξί κλικ στον κλάδο “feature/123” και να τον σπρώξετε ανεξάρτητα. Μπορείτε να αποθηκεύσετε τα πρόσθετα βήματα της αποθήκευσης, του ελέγχου αυτού του κλάδου, της ώθησης, του ελέγχου του προηγούμενου υποκαταστήματος ξανά και της απόρριψης της αποθήκευσης.

Αναίρεση και Επανάληψη

Η μη αυτόματη αναίρεση λειτουργιών Git είναι περίπλοκη και φαίνεται σχεδόν αδύνατη σε ορισμένες περιπτώσεις. Με το GitKraken Client είναι εύκολο: απλώς κάντε κλικ στα κουμπιά «Αναίρεση» ή «Επανάληψη» στη γραμμή εργαλείων ή χρησιμοποιώντας την Παλέτα εντολών.

Αναίρεση: Ο client GitKraken μπορεί να αναιρέσει την checkout, να δεσμεύσει, να απορρίψει, να διαγράψει ένα branch , να αφαιρέσει ένα remote ή να επαναφέρει έναν branch σε commit με ένα μόνο κλικ του κουμπιού «Αναίρεση» (ή, μπορείτε να το κάνετε μέσω της Παλέτας εντολών).

Redo: Έκανες κάτι και μετά το αναίρεσες, μόνο και μόνο για να ευχηθείς να το είχες αφήσει μόνο του; Υπάρχει μια λειτουργία “Redo” για αυτό!

Pull Requests

Με την ενσωμάτωση του GitKraken Client με το GitHub, μπορείτε να δημιουργήσετε, να προβάλετε, ακόμη και να κλείσετε και να συγχωνεύσετε PR χωρίς να αφήσετε τα αγαπημένα πλοκάμια του GitKraken.

Εάν χρησιμοποιείτε τις ενσωματώσεις GitLab ή GitHub, μπορείτε επίσης να προσθέσετε  pull request assignee και label στο PR. Στη συνέχεια, το GitKraken Client θα μεταβιβάσει αυτές τις αλλαγές στο GitLab ή στο GitHub όταν δημιουργηθεί το PR.

Επιπλέον, για την ενσωμάτωση του GitLab, μπορείτε να τοποθετήσετε το δείκτη του ποντικιού πάνω από ένα υπάρχον PR για να εμφανίσετε τυχόν assignes ή labels που σχετίζονται με το αίτημα έλξης. Και για το GitHub, αυτή η επεξήγηση εργαλείου θα εμφανίζει  τους assignees, τα labels, τους reviewers, και το build status.
